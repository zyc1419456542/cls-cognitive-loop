# CLS 认知循环 — 真实案例

> 三个可公开案例，展示 CLS 6步认知循环 + 双AI闸门在不同任务类型中的实际表现。
> 所有案例已脱敏，不包含真实姓名、公司名称、API端点或客户信息。

## 案例一：CAD 多零件装配管线

### 任务背景

设计一套 15 零件组成的霍尔推力器装配体（壳体、内外磁极、阳极、气体分配器、绝缘件），需满足包含磁路间隙、气体拓扑路径和热间隙在内的约 40 条机械约束。传统流程中，设计师在单次 LLM 交互中生成 CAD 脚本，然后用 CAD 软件打开 STEP 文件手动检查——这个过程会迭代 5-8 轮，每次迭代都丢失先前的约束上下文。

### CLS 如何介入

| 认知步骤 | 实际动作 |
|---------|---------|
| ① 态势感知 | 读取知识库中的三组约束文档（磁路约束注册表、气体拓扑规则、螺栓间隙标准），识别为"多约束 CAD 装配"任务 |
| ② 任务执行 | 判定为复杂任务（15 零件, 40+ 约束），生成 build123d 脚本，每10个约束跑一次约束图验证 |
| ③ 关联学习 | 发现新约束："当螺栓直径与壁厚同量级时，必须检查通体间隙"——该模式此前未在知识库中记录 |
| ④ 抽象泛化 | 抽象为可迁移模式：螺栓直径/壁厚 > 0.6 则需通体间隙检查，补充适用条件和反例 |
| ⑤ 上下文持久化 | 更新磁路约束文件和气体拓扑规则文件，确保下次类似任务自动加载 |
| ⑥ 轨迹更新 | 记录本轮设计演化路径（零件添加顺序、约束检查时间线、异常修复记录） |

双AI闸门在每次 STEP 导出前运行：DeepSeek 生成装配脚本 → Qwen 独立验证几何合理性和约束完整性 → 验证未通过则自动回到步骤②重新生成。

### 与传统方法对比

| 维度 | 传统单次LLM | CLS 认知循环 |
|------|------------|-------------|
| 设计迭代 | 5-8 轮 | 1-2 轮 |
| 上下文丢失 | 每次迭代150-300 条消息后丢失 | 每次迭代自动持久化 |
| 约束验证 | 人工视觉检查 | 自动约束图+双AI验证 |
| 知识积累 | 无 | 每轮提取1-3条新模式 |
| 错误发现时机 | STEP导出后人工检查 | 生成中实时拦截 |

### 量化收益

- 设计迭代次数减少 **70%**（从 6.5 轮平均降至 2 轮）
- 首次设计正确率从 40% 提升到 **85%**
- 约束违规在 STEP 导出前被自动捕获，零次因约束问题返工
- 知识库累计沉淀约束模式 40+ 条，后续装配任务可直接复用

## 案例二：教辅试卷双AI审计管线

### 任务背景

生成一套完整的高考模拟数学卷解答（12 题，含详细步骤推演和评分标准），涵盖多难度层级。传统方式中，单模型生成后自我检查——但同一模型重复犯相同的系统性错误（概率误判、参数端点遗漏、不等式方向反转等），错误率约 12%。

### CLS 如何介入

验证端系统提示词中注入了 5 类已知错误模式：
1. **马尔可夫链路径概率** — 不假设等可能，逐乘条件概率
2. **椭圆参数方程** — 统一用定比分点而非截距式分类
3. **数列范围端点** — 递推时检查定义域排除退化点
4. **复数三角不等式** — 模长最值用 ||u|-|v|| 降自由度
5. **坐标代入纪律** — 验证代入值在曲面/曲线上而非外推

流程：Generator (DS) 生成解答 → Evaluator (Qwen) 独立验证，逐题对照标准答案和已知错误模式 → FAIL 时熔断板拦截，带具体修改指令回到生成端 → 重试上限 2 次 → 仍 FAIL 则人工介入。

### 与传统方法对比

| 维度 | 单模型自检 | CLS 双AI闸门 |
|------|----------|-------------|
| 错误率 | ~12% | **~1%** |
| 系统性错误来源 | 同一模型的认知盲区 | 两个独立模型同时犯同一错的概率 = 乘积 |
| 验证独立性 | 无（模型自评） | 独立模型+独立上下文 |
| 已知错误模式注入 | 无 | 自动注入5+类模式 |
| 人工复核负担 | 高（逐题检查） | 低（只审核 FAIL 项） |

### 量化收益

- 整体错误率从 12% 降至 **约 1%**（符合 P(DS错) × P(QW错) ≈ 10% × 10% = 1% 的理论预期）
- 已交付 6 套试卷，**零未检测错误**
- 平均审计周期：3-5 分钟/套（含最多 2 轮自动修正）
- 人工审核从逐题检查变为只关注闸门拦截项

## 案例三：跨领域复杂项目交付

### 任务背景

一个多阶段技术项目，涵盖需求分析、算法设计、仿真验证和技术文档撰写四个阶段。传统方式需要至少 3 名领域专家（系统工程师、算法工程师、仿真工程师）在 2 周内串联完成，每次阶段交接丢失 15-30% 的上下文。

### CLS 如何介入

任务被分解为 **6 个完整认知周期**：

| 周期 | 阶段 | 认知循环产出 |
|------|------|------------|
| C1-C2 | 需求分析 | 态势感知扫描已有需求库 → 关联学习匹配历史类似项目 → 抽象泛化提取需求模式 → 固化需求文档和验收标准 |
| C3-C4 | 设计+仿真 | 任务执行生成方案 → 双AI闸门验证设计逻辑和仿真参数 → 关联学习沉淀新设计规则 |
| C5-C6 | 文档生成 | 并行跨窗口协调三个工作流（算法文档、验证报告、用户手册）同时生成和互审 |

关键机制：跨窗口感知（三个文档工作流并行），事实锚定（所有技术声明绑定到具体数据文件路径），熔断板 Token 预算和并行上限防失控。

### 与传统方法对比

| 维度 | 传统瀑布 | CLS 认知循环 |
|------|---------|-------------|
| 阶段交接上下文损失 | 15-30% | **<5%**（自动持久化） |
| 专家人数 | 3人 | 1人监督 |
| 总耗时 | 14天 | **3天** |
| 人机交互模式 | 操作员（每步指令） | 监督员（仅关键决策介入） |
| 交付物一致性 | 阶段间可能矛盾 | 跨阶段事实锚定保证一致性 |

### 量化收益

- 交付周期缩短 **78%**（14天 → 3天）
- 人工干预从全程操作降为 2 次关键决策点
- 所有验证闸门通过，交付物零阶段间矛盾
- 项目全部轨迹和决策记录可追溯，支持事后审计

## 三个案例的共同线索

1. **短任务+独立验证+串联成环** — 单次 LLM 交互的不确定性被结构化为可验证的闭环
2. **人类角色转型** — 从操作员到监督员，只在闸门熔断时才介入
3. **知识持续积累** — 每次循环都在沉淀模式，后续任务启动时知识库自动加载，非从零开始
4. **安全性内生** — 熔断板和双AI闸门不是事后补救，是系统的基础架构层
