# CLS 认知循环系统

> **一个将 LLM Agent 推理结构化为可验证 6 步闭环的 Harness 架构 —— 配备双模型验证、纯 stdlib 熔断板、跨会话持续学习。**

*作者：**认知工坊** · QQ 1419456542 · zyc2018@mail.ustc.edu.cn*

---

## CLS 解决什么问题？

**LLM Agent 做长任务必然跑偏。** 每一步单独看合理，但整体一定偏离——因为模型没有稳定的状态系统。`prompt + history` 只是文本序列，会膨胀、会自指、会遗忘早期约束。

**CLS 给 Agent 装了个"方向盘"。** 每个任务强制经过 6 步闭环（态势感知→执行→学习→泛化→持久化→轨迹更新），每一步之间的状态是可验证的，跨 session 也不会失忆。

**独立起源。** 首版代码 commit [`73ee3a84`](https://github.com/zyc1419456542/cls-cognitive-loop/commit/73ee3a84) 于 **2026 年 5 月 25 日**落地——比 6 月 7 日 Google Addy Osmani 提出 "Loop Engineering" 概念早了约 2 周。

## CLS 和其他方案有什么区别？

| 你可能想到的 | 它做什么 | CLS 多了什么 |
|------------|---------|------------|
| **直接调 API**（ChatGPT、Claude 等） | 单次生成，无状态 | 6 步结构化闭环，每步之间有状态检查、安全校验、自我修正 |
| **LangChain / LlamaIndex** | 工作流编排 + RAG | 认知循环（不是管线）、双 AI 验证（不是自评）、纯 stdlib 熔断板（不是软约束） |
| **AutoGPT / BabyAGI** | 自主任务拆解 | 每轮固定 6 步 —— 不无限生成子任务。稳定性和可验证性优先于探索性 |
| **Loop Engineering**（Google/Anthropic，2026.6） | 循环式 Harness 的*概念* | CLS 在该概念被命名前 2 周已有可运行代码。双 AI 闸门（生成/验证分离）和熔断板（纯 stdlib 安全层）为 CLS 独有 |
| **Claude Code / Cursor** | 带 Agent 能力的开发工具 | 一个可集成到任何宿主机的 Harness *架构*，不是独立产品 |

## 30 秒看懂架构

```
┌──────────────────────────────────────────────────────┐
│ ① 态势感知　→　② 任务执行                             │
│        ↑                    ↓                         │
│ ⑥ 轨迹更新　←　③ 关联学习                             │
│        ↓                    ↓                         │
│ ⑤ 上下文持久化　←　④ 抽象泛化                          │
└──────────────────────────┬───────────────────────────┘
                           │
           ┌───────────────▼───────────────┐
           │ 熔断板（6个独立熔断器）         │
           │ 双 AI 闸门（DS→Qwen 验证）     │
           │ 符号动力学审计                 │
           └───────────────────────────────┘
```

**6 步闭环（每步一句话）：**

| 步骤 | 名称 | 核心逻辑 |
|------|------|---------|
| ① | 态势感知 | 读状态文件、感知其他会话窗口、动手前搞清楚自己在哪 |
| ② | 任务执行 | 规模判断→Pre-flight 检查（熔断板、前提验证）→执行→验证 |
| ③ | 关联学习 | 新知识与已有模式匹配，标记矛盾 |
| ④ | 抽象泛化 | 具体经验提炼为可迁移的模式 |
| ⑤ | 上下文持久化 | 写检查点到磁盘——崩了能续上，跨 session 不丢 |
| ⑥ | 轨迹更新 | 记录问题怎么演化的（Δq）和进展怎么变化的（Δp） |

闭环：⑥ → ①。不是线性流水线，是每轮更紧的循环。

## 核心差异点

### 1. 独立起源（时间线证据）

CLS 来自生产实践，不是来自理论推演。时间线可验证：

| 日期 | 事件 |
|------|------|
| **2026-05-25** | CLS 6 步认知循环首次提交代码（`73ee3a84`） |
| 2026-06-02 | 熔断板（6 个独立熔断器）激活 |
| 2026-06-03 | 双 AI 闸门（生成/验证分离）强制执行 |
| 2026-06-07 | Addy Osmani 发表《Loop Engineering》博文 |
| 2026-06 中旬 | Boris Cherny 与 Peter Steinberger 公开讨论 loop 式 harness |

**这意味着：** CLS 没有跟随趋势——它在行业公开讨论同一洞察的 2 周前就已经在跑了。架构是经过实战检验的，不是理论产物。

### 2. 双 AI 闸门（生成/验证分离）

一个模型永远不应该评估自己的产出。CLS 强制执行：

- **DeepSeek** 负责创造（代码、设计、知识）
- **Qwen** 独立验证（正确性、安全性、一致性）

统计学保证：P(系统错) = P(DS 错) × P(QW 错)。单端错误率 10% 时，双重错误率 ≈ 1%。Qwen 不可用时降级到 Haiku。

### 3. 熔断板（纯 stdlib 熔断器）

6 个独立熔断器，**纯 Python stdlib 编写**——不 import 任何受保护的认知模块。被保护的系统无法篡改它。

| 熔断器 | 防止什么 | 触发行为 |
|--------|---------|---------|
| WRITE_PROTECT | 自改核心文件 | 阻止对保护路径的写入 |
| RECURSION_LIMIT | 无限自指嵌套（最大深度 5） | 截断 + 30 分钟冷却 |
| TOKEN_BUDGET | API 费用失控（日 200 万/会话 50 万） | 阻止 API 调用 |
| PARALLEL_CAP | 同时过多破坏性操作（最多 3） | 阻塞新并行任务 |
| CHECKPOINT_REQUIRED | 重大变更前无回滚点 | 每 300 秒强制保存 |
| PROXY_PURITY | 代理层语义变换 | 只允许删除字段 |

`FuseBackend` 抽象接口意味着今天软件实现，明天可以换成硬件实现，调用代码不变。

### 4. 符号动力学 & 跨窗口感知

- **符号审计：** 5 万 token 对话 → 3 个数字（拓扑熵、谱半径、告警计数）+ 1 个状态词。每次检查约 200ms。
- **跨窗口感知：** 多个 Claude Code 会话共享状态文件，互相感知焦点，避免领域冲突。
- **事实锚定：** 每个声明必须引用具体的文件路径和字段值。无证据的陈述在结构上被拒绝。

## 快速开始

### 前提

- Claude Code（或任何 LLM Agent 宿主——架构与宿主无关）
- Python 3.10+
- （可选）为双 AI 闸门准备第二个 LLM API

### 快速验证

```bash
# 熔断板自检（纯 stdlib，无需外部依赖）
python scripts/safety/fuse_board.py --test

# 符号动力学引擎示例数据运行
python scripts/safety/symbolic_dynamics_engine.py --test

# 双 AI 闸门健康检查（需要 API Key）
python scripts/safety/qwen_gate.py --health
```

### 设计哲学（三点）

1. **接口优先于实现** — 签名固定，后端可换。今天软件熔断器，明天硬件熔断器，同一套 API。
2. **三元乘法，不是 AGI** — CLS = AI × CLS × 人类。任何因子为零结果即为零。不是通用智能，而是稳定的闭环。
3. **稀疏激活，共享基线** — 安全层始终在线。领域技能按需加载。路由失败时基线仍然可用。

## 为什么开源？

CLS 是一个月生产开发的产物——为解决真实问题（CAD 设计、教辅审计、跨领域交付）而建，不是研究项目。架构已在实践中验证。开源意味着接受审视、接受贡献、接受独立验证。

> *"循环不提升模型的智商。它提升的是系统的稳定性、复用率和长期积累。短任务、独立验证、串联成环——这就是架构。"*
